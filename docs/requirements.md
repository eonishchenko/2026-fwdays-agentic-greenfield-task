# Requirements

Джерело: [prd_draft.md](./prd_draft.md).  
Продукт: покроковий генератор рахунків і актів з шаблонів, збереженням контактів/послуг і експортом у PDF/JSON.

---

## FR — Functional Requirements

| ID | Вимога | Пріоритет |
|----|--------|-----------|
| FR-01 | Система створює документний сеанс з унікальним `guid` і зберігає стан у `./docs/{guid}.json`. | Must |
| FR-02 | Постійний URL сеансу: `/docs/{guid}`. Відкриття URL відновлює стан і веде на останній незаповнений крок або на фінальний перегляд, якщо флоу завершено. | Must |
| FR-03 | Користувач може обрати тип документа: «рахунок + акт» (default), «рахунок», «акт». | Must |
| FR-04 | Користувач може скопіювати дані з існуючого `guid` (усі поля крім номерів документів). | Should |
| FR-05 | На кроці дати/номерів: дата за замовчуванням — поточний день; редагування дати дозволене. | Must |
| FR-06 | Номер рахунка генерується як `Р-{NNNNNNN}` (7 цифр). Номер акта — послідовне ціле `{N}`. | Must |
| FR-07 | Останній згенерований номер зберігається в `./{year}/doc_number.json`, де `year` — рік дати документа. | Must |
| FR-08 | Користувач обирає або створює контакт замовника. Контакт зберігається за РНОКПП у `./contacts/{РНОКПП}.json`. | Must |
| FR-09 | Користувач обирає або створює контакт виконавця з тією ж схемою збереження, що й FR-08. | Must |
| FR-10 | Поля контакту: `full-name`, `inn` (РНОКПП), `phone`, `acc`, `bank`, `mfo-bank`, `addr`. | Must |
| FR-11 | Користувач обирає послуги з каталогу або створює нові. Каталог зберігається в `./jobs/services.json`. | Must |
| FR-12 | Поля послуги: `sign-name`, `service-name`, `amount`, `price`. Можна додати кілька послуг у документ. | Must |
| FR-13 | Система заповнює шаблон рахунка `./templates/invoce.html`: замовник (`client-*`), виконавець (`done-by-*`), `amount-cursive`, `date`, `date-cursive`, `service-cursive`, `done_amount`, `price`, `price-total`. | Must |
| FR-14 | Система заповнює шаблон акта `./templates/complete.html`: замовник (`client-*`), виконавець (`done-by-*`), `amount-cursive`. | Must |
| FR-15 | Фінальний крок показує сформовані документи без редагування вмісту; для кожного — завантаження PDF; є повернення на попередній крок. | Must |
| FR-16 | З екрана перегляду документа доступні: завантаження PDF і «Редагувати» (повернення у флоу створення). | Must |
| FR-17 | На кожному кроці флоу є навігація «Назад» / «Далі» (окрім обмежень першого/останнього кроку). | Must |
| FR-18 | UI показує прогрес кроків у вигляді номерів у кружках. | Must |
| FR-19 | Завершений сеанс можна експортувати/зберегти як JSON-стан документа (`./docs/{guid}.json`). | Must |
| FR-20 | Helper: конвертація суми в пропис (uk, гривня). Приклад: `8750` → `Вісім тисяч сімсот п'ятдесят гривень 00 копійок`. | Must |
| FR-21 | Helper: конвертація дати в пропис. Приклад: `15.05.2021` → `15 травня 2021 р.`. Числовий формат дати в шаблоні: `DD.MM.YYYY`. | Must |

### Кроки флоу

| Крок | Назва | Поведінка |
|------|-------|-----------|
| 1 | Тип документа | Тип або копіювання з `guid` |
| 2 | Дата і номери | Дата, `Р-#######`, номер акта; інкремент лічильника року |
| 3 | Замовник | Вибір / створення контакту |
| 4 | Виконавець | Вибір / створення контакту |
| 5 | Послуги | Каталог + рядки документа |
| 6 | Заповнення шаблонів | Preview заповненого HTML (рахунок і/або акт за типом) |
| 7 | Готові документи | Read-only + PDF + назад |

### Поля шаблону рахунка (крок 6)

| Поле / префікс | Зміст | Приклад |
|----------------|-------|---------|
| `client-*` | Замовник | — |
| `done-by-*` | Виконавець | — |
| `amount-cursive` | Сума прописом | `Вісім тисяч сімсот п'ятдесят гривень 00 копійок` |
| `date` | Дата `DD.MM.YYYY` | `15.05.2021` |
| `date-cursive` | Дата прописом | `15 травня 2021 р.` |
| `service-cursive` | Послуга (текст для документа) | — |
| `done_amount` | Кількість | — |
| `price` | Ціна | — |
| `price-total` | Підсумок сум усіх позицій | — |

## NFR — Non-Functional Requirements

| ID | Категорія | Вимога |
|----|-----------|--------|
| NFR-01 | UX | Нейтральна палітра, доречна для фінансових документів (без «маркетингового» UI). |
| NFR-02 | UX | Прогрес флоу завжди видимий; поточний крок виділений. |
| NFR-03 | Persistence | Стан сеансу переживає перезавантаження сторінки через `guid` і файлове сховище. |
| NFR-04 | Idempotency | Повторне відкриття `/docs/{guid}` не створює новий `guid` і не змінює номери без явної дії користувача. |
| NFR-05 | Data integrity | Контакти ключуються унікальним РНОКПП; оновлення контакту перезаписує той самий файл. |
| NFR-06 | Numbering | Лічильники номерів атомарні в межах року; різні роки мають незалежні файли `doc_number.json`. |
| NFR-07 | Templates | PDF візуально відповідає HTML-шаблонам A4 (`invoce.html`, `complete.html`). |
| NFR-08 | Locale | Інтерфейс, сума прописом і дата прописом — українською; валюта — гривня. |
| NFR-09 | Performance | Перехід між кроками < 300 ms на локальних даних (без мережевих затримок). |
| NFR-10 | Reliability | Помилка збереження контакту/послуги/сеансу показується користувачу; частково збережений стан не «мовчки» губиться. |
| NFR-11 | Security (MVP) | Немає публічної автентифікації; доступ за знанням `guid` (security-through-obscurity прийнятний для локального/внутрішнього MVP). |
| NFR-12 | Portability | Шляхи даних відносні до кореня даних застосунку; шаблони читаються з `./templates/`. |
| NFR-13 | Naming | Ключі JSON і CSS-якорі шаблонів — kebab-case (`full-name`, `mfo-bank`, `done-by-*`), окрім явно зафіксованих у PRD (`done_amount`). |

## BC — Business Constraints & Boundaries

| ID | Тип | Обмеження |
|----|-----|-----------|
| BC-01 | Scope | MVP покриває лише рахунок і акт надання послуг (ФОП ↔ ФОП), без договорів, накладних, ПДВ-рахунків окремого типу. |
| BC-02 | Scope | Немає мультикористувацького акаунтингу, ролей і аудиту змін. |
| BC-03 | Scope | Немає інтеграцій з банком, ДПС, ЕДО, email-розсилкою. |
| BC-04 | Data | Джерело істини — локальна файлова система (JSON + HTML-шаблони), не зовнішня БД. |
| BC-05 | Data | Ідентифікатор контакту — РНОКПП; дублікати з різним написанням ФІО при тому ж РНОКПП вважаються одним контактом. |
| BC-06 | Numbering | Формат номера рахунка фіксований: префікс `Р-` + 7 цифр. Формат акта — ціле без префікса. |
| BC-07 | Templates | Імена файлів шаблонів фіксовані: `invoce.html` (історична назва), `complete.html`. |
| BC-08 | Legal | Система не є бухгалтерським ПЗ і не гарантує відповідність усім вимогам законодавства; користувач відповідає за зміст. |
| BC-09 | Access | Знання URL/`guid` достатнє для доступу до документа (MVP). |
| BC-10 | Locale | Підтримується лише українська локаль і гривня для суми/дати прописом. |
| BC-11 | Copy | При копіюванні з іншого `guid` номери документів завжди генеруються наново. |
| BC-12 | Edit | Після фіналізації вміст HTML на кроці 7 не редагується inline; зміни — лише через повернення у флоу. |
| BC-13 | Helpers | Формати пропису суми й дати фіксовані прикладами з PRD; інші локалі/валюти не підтримуються. |
